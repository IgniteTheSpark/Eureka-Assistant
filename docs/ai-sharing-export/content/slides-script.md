## Slide 3. 我一开始以为:AI Coding = 更快写代码

**页面重点:** 先打破“Prompt -> Code”的简单理解。

**口播:**

我最开始对 AI Coding 的理解也很直接:我描述需求,AI 帮我写 Flutter 页面、FastAPI 接口、测试和 bug fix。  
这些当然有用,但做 UReka 后我发现,代码其实是最后一步。

如果产品方向没有想清楚,AI 会很快帮你做出一个“不太对但看起来完整”的东西。  
如果数据边界没想清楚,AI 会把页面、接口、状态都写出来,但后面一连真实场景就开始返工。  
所以我现在理解的 vibe coding,不是更快写代码,而是更快完成“提出假设、做出版本、验证不对、再调整”的循环。

## Slide 4. 起点:我想放大“闪念”

**页面重点:** 讲清楚为什么做 UReka,不是为了做一个记录 App,而是为了放大“闪念”能力。

**口播:**

UReka 的起点不是“我要做一个 App”,而是我一直想放大“闪念”这个动作。  
普通人每天有很多很碎的内容想记录:花了多少钱、孩子喝了多少奶、突然想到一个产品点子、今天运动了什么、读书看到一句话、工作里一个待办。

问题不是大家不会记录,而是入口太重、后续太散。  
很多内容散落在不同 App 里,或者最后都堆进备忘录,过几天就很难再被整理和利用。  
所以我的核心假设是:如果有一个统一输入入口,再配合一个统一的 AI 整理层,碎片记录就有机会变成可沉淀的个人资产。

硬件闪念按钮、戒指入口、App 里的闪念会话,本质上都在服务同一个问题:让“我想记一下”这个动作足够轻。

## Slide 5. UReka 是什么:统一入口 + AI 整理层

**页面重点:** 用日历、资产库、今日页三张静态图解释产品表面。

**口播:**

这一页可以把 UReka 理解成三层体验。  
第一层是输入:用户可以说一句、按一下,或者在当天页面里记一笔。  
第二层是 AI 整理:系统理解这句话是记账、事件、待办、灵感,还是某个自定义技能,然后归类成结构化资产。  
第三层是输出:这些资产不只是存在数据库里,还会进入日历、资产库、Today 页、提醒和报告。

这里我放了三张静态图:日历页、资产库、今日页。  
日历负责把记录放回时间语境,资产库负责长期沉淀和分类,今日页负责每天打开时给用户即时反馈和下一步建议。  
所以 UReka 不是一个单点功能,而是一个从输入到整理再到反馈的闭环。

## Slide 6. UReka 的技术架构

**页面重点:** 不只说用了什么,重点讲为什么要这样分层。

**口播:**

技术架构这里,我想重点讲“为什么需要这些层”,而不是报技术栈。  
UReka 如果只是一个聊天 App,一个前端加一个 LLM API 就够了。  
但它要做的是长期记录、结构化资产、主动提醒、报告生成、硬件入口和第三方集成,所以必须把系统拆成几个稳定边界。

最上层是 Flutter App。原因很简单:UReka 是高频移动体验,很多交互和状态都发生在手机上,比如 Today、Flash、Ring、Reports 和 Reka 宠物。  
它需要足够原生,也需要能承载动画、离线状态、设备连接和 WebView 报告。

第二层是 FastAPI Backend。它承担的是产品系统的确定性部分:用户认证、资产 CRUD、事件、报告、通知、connected apps。  
这些东西不能都丢给 Agent,因为它们需要权限、事务、可追踪和稳定接口。

第三层是 ADK Agents + Pipelines。这里是 AI 真正发挥作用的地方,但我没有把它设计成一个万能 Chat。  
这里的 ADK 可以简单理解成 Agent Development Kit,也就是搭建 Agent 的开发框架。  
它帮我把模型、工具、状态、步骤编排这些东西组织起来,不需要每次都从一个裸的 LLM API 开始写。  
Flash pipeline 处理闪念结构化,Chat agent 处理对话和工具调用,Report pipeline 处理报告生成,Task pipeline 处理异步任务。  
拆成 pipeline 的原因是:每条链路的输入、工具、验证标准都不同,拆开后更容易 debug,也更容易让 Agent 做对。

第四层是 FastMCP + External MCP。MCP 的价值是把工具边界显式化。  
MCP 可以理解成 Agent 调用外部能力的一种协议。  
如果说模型负责“想”和“判断”,那 MCP 暴露的工具就负责“做事”:查数据库、写资产、读日历、发通知、接 DingTalk 或 Notion。  
FastMCP 则是我用来快速把内部业务能力包装成 MCP 工具的方式。  
Agent 需要知道自己能做什么、不能做什么,比如内部 CRUD、DingTalk、Notion、Calendar 这些工具都应该是明确的 capability,而不是在 prompt 里随便写。

最后是 MySQL + Memory。记录产品最重要的是沉淀,所以资产、事件、报告、nudges、user skills 都要进入结构化数据层。  
Memory 不是抽象概念,它是下一次 Agent 判断时能依赖的上下文来源。  
比如用户过去经常记录网球、宝宝饮食、消费或工作灵感,这些历史结构会影响下一次 Agent 如何理解一句新的闪念。  
所以这套架构的核心不是“堆技术”,而是把不确定的 AI 能力放在 pipeline 和 tool boundary 里,把确定性的业务状态放在后端和数据层里。

## Slide 7. 我的 Agent 开发工作流

**页面重点:** 把 spec library 截图放进工作流,强调不是边聊边写,而是阶段化协作。

**口播:**

做到后面我发现,vibe coding 不能只是边想边让 AI 写。  
更稳定的方式是把 Agent 放在不同阶段,每个阶段有不同目标和产物。

第一步是 Brainstorm 或 Office Hours,把模糊想法变成产品问题。  
第二步是 Writing Plans 和 Specs,把需求拆成边界、任务、风险和验证标准。右边的 spec library 就是这个过程留下的资产,不是聊完就算了。  
第三步是 Design Exploration,用设计参考和产品判断探索界面方向。  
第四步才是 Coding Agents,让独立 Agent 根据任务实现。  
后面还要做 auto review、manual review,再进入 bug fix loop。

这里的重点是:每一步都要有产物。  
brainstorm 的产物是方向,writing plan 的产物是 spec,design 的产物是可判断的界面方案,coding 的产物是可运行代码,review 的产物是可修复问题。  
这样 Agent 才不是一个随叫随到的打字机,而是一个可协作的产品开发系统。

## Slide 8. Case 1:日历页面如何被重做三次

**页面重点:** 用三版演进说明 Agent 不是一次给对答案,而是帮助快速推翻。

**口播:**

第一个 case 是日历页。  
第一版是传统顶部日期模式,很快能做出来,也很容易看懂,但问题是太普通了。  
它像一个普通日程列表,没有 UReka 自己的记忆感,也没有把“碎片记录”和“时间语境”结合起来。

第二版参考了 Timepage。  
这一版视觉和动效方向明显更强,滚动体验也更接近我想要的感觉。  
但很快又出现一个问题:动效交互可以了,内容展示还是不对。  
UReka 不只有日程,还有记账、待办、灵感、外部引用、自定义技能记录。单纯做一个漂亮日历并不能解决内容结构问题。

第三版就是现在这个版本。  
它把日程、非日程、分类记录和当天语境放在一起,不是为了炫交互,而是让用户能在一天里看到“发生了什么、记录了什么、还有什么没处理”。  
这个过程很典型:AI 一开始也会做出很像样的界面,但你必须不断追问:它解决的是视觉问题,还是产品问题?

## Slide 9. Case 2:首页交互 - 主动建议和资产反馈

**页面重点:** Tinder Page 和球球池分别解决“下一步行动”和“记录正反馈”。

**口播:**

第二个 case 是首页交互。  
这里我最后保留了两个机制:Reka Offer 的 Tinder Page,以及底部的球球池。

Tinder Page 解决的是“下一步行动”的问题。  
如果系统发现用户有逾期任务、积累了很多消费记录、或者某些记录可以生成报告,它不应该只放在一个列表里等用户找。  
Reka Offer 会主动把建议推到用户面前,用户可以左滑跳过、右滑执行、点进去看详情。  
这个交互的价值是把 AI 建议变成低成本决策。

球球池解决的是“记录后的正反馈”。  
用户说一句、按一下,系统把它变成资产,如果只是安静地存进数据库,用户感受不到价值。  
球球掉落让用户看到“我刚刚创造了一个东西”,这会强化记录行为。  
所以这两个交互一个负责主动建议,一个负责资产反馈,共同让首页从信息展示变成了一个每天会发生事情的地方。

## Slide 10. Agent Design:从 App 功能到 Agent 系统

**页面重点:** 解释 flash pipeline 为什么已经是局部 Agent loop,并澄清 cron/heartbeat 的边界。

**口播:**

这里有一个很重要的认知:UReka 不是“用户说一句,AI 回一句”。  
Flash pipeline 本身已经是一种局部 Agent loop。  
它收到自然语言输入后,要判断类型,选择 skill 或 parser,抽取结构化字段,写入资产,再验证结果是否能用于后续展示和报告。

这里的 Input、Reason、Act、Verify、Memory 不是抽象图。  
Input 是用户说的一句话或硬件触发。  
Reason 是判断这条记录属于什么领域、需要哪些字段。  
Act 是调用工具写入资产、事件或报告任务。  
Verify 是检查字段是否完整、是否需要追问。  
Memory 是把这次记录沉淀为下一次判断的上下文。

另外,cron 和 heartbeat 是更主动的系统能力。  
cron 可以是“到某个时间点检查是否应该生成日报、周报、提醒”。  
heartbeat 更像持续轻量扫描:比如用户今天是否还没有记账,某个待办是否拖了太久,灵感是否积累到可以整理。  
这些不是为了让系统显得复杂,而是为了让 UReka 从被动记录工具,慢慢变成主动的个人记录 Agent。

## Slide 11. 硬件戒指接入:把闪念按钮戴在手上

**页面重点:** 强调硬件戒指是低摩擦入口,不是硬件炫技。

**口播:**

硬件戒指接入这页,我想强调的不是“我们有一个硬件”。  
真正重要的是:记录入口能不能轻到接近肌肉记忆。

很多时候人不是不想记录,而是当下打开手机、找 App、选入口、打字这几步太重了。  
戒指按钮的价值是把“闪念来了”变成一个稳定的物理触发:按一下,先把它捕捉下来。  
后面的 ASR、理解、分类、资产写入,都可以交给系统慢慢完成。

这里 AI 帮到我的地方也不只是写界面。  
它要帮我拆连接链路、录音状态、异常恢复、后端接收、ASR 结果和资产写入。  
这类 case 让我意识到,AI coding 一旦碰到硬件和真实状态,就必须回到工程链路本身,不能只停留在 UI。

## Slide 12. AI 改变的是迭代密度

**页面重点:** commit + branch 截图说明高频试验和拆分方向。

**口播:**

这一页是 GitHub 的 commit 和 branch。  
我想表达的是:AI 带来的不是“一次生成一个完整产品”,而是迭代密度变高。

你可以看到这里不只是 implementation,还有 spec、branch、verify、bug fix。  
branch 的意义是把不同方向拆开:calendar redesign、today home redesign、flash pipeline、optimization。  
每个方向都可以有独立 Agent 去推进,也可以被单独 review 和回滚。

以前一个人做产品,从想法到设计到实现到验证,周期会比较长。  
现在更像是一天里可以并行推进多个小试验:一个 Agent 写 spec,一个 Agent 做页面,一个 Agent 修 bug,我自己做最终判断。  
所以 AI 改变的不是“我不用思考了”,而是我能更快地看到多个可能性,然后更快做取舍。

## Slide 13. 生成的不是内容,而是体验

**页面重点:** 报告列表 + 报告详情两种 GIF,说明生成内容还需要产品化表达。

**口播:**

报告这一页我放了两种状态:报告列表和报告详情。  
这其实是我对“AI 生成内容”的一个判断:生成一段总结不难,但让用户愿意回来看、读得进去、读完知道下一步做什么,才是产品体验。

报告列表解决的是管理和回看。  
用户需要知道最近有哪些报告,哪些是晨间简报,哪些是灵感综合,哪些是消费复盘。  
报告详情解决的是阅读体验。  
它不能只是一段 markdown,而要有标题、结构、引用、重点、视觉层级和情绪。

所以 report pipeline 不是简单地把历史记录扔给模型。  
它要先确定报告类型和范围,拉取真实数据,生成结构化内容,再用确定性渲染器做成可读页面。  
AI 让内容生成变便宜,但好的表达仍然需要设计。

## Slide 14. 审美是 AI 时代的输入能力

**页面重点:** Variant、Craftwork、Hugeicons 作为审美输入来源。

**口播:**

过去我们说审美,容易觉得它是主观感觉。  
但在 AI 工作流里,审美其实是一种输入能力。  
你看过什么、能不能拆解为什么好、能不能把它转成明确设计语言,会直接影响 AI 的产出质量。

比如 Variant 给我的是生成式设计和 remix 的感觉,Craftwork 给我的是更丰富的真实产品参考,Hugeicons 给我的是图标语言和细节密度。  
这些不是截图堆砌,而是告诉 Agent:我要的不是“好看一点”,而是某种信息密度、视觉层级、动效节奏和组件气质。

所以后面我会结合 taste skill、GSAP skill、web-to-design 这类工具,把审美参考转成更明确的约束。  
AI 很容易产出平均值,而审美输入决定了它有没有可能跳出平均值。

---

## Slide 15. 总结:vibe coding 不只是写代码

**页面重点:** 收束为方法论和踩坑。

**口播:**

最后总结一下。  
这个过程里我最大的体会是:vibe coding 不只是写代码,而是把产品判断、Agent 架构、工程实现、质量验证和表达能力串起来。

AI 会迎合你,所以要反复问:这是不是核心价值?  
AI 会过度实现,所以要用 spec 收边界。  
上下文会漂,所以要用文档、handoff 和 commit 稳住。  
代码能跑不等于产品对,日历页和首页交互都是反复推翻后才靠近正确方向。  
设计很容易 generic,所以需要参考和 taste 输入。

如果只把 AI 当成写代码工具,它当然能提效。  
但如果把它当成一个可以参与产品、架构、设计、实现、验证的协作系统,它改变的就是一个人做产品的方式。
