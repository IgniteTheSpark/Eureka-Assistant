## 11.0 定位与铁律

- **纯内部工具,绝不对终端用户暴露。** 独立 admin 鉴权 + 角色门控,与普通用户登录完全隔离(§10.3 安全铁律)。
- **沿用 §10.0 边界**:代码拥有「画法 + 指标计算 + schema 种类」,后台拥有「元数据 + 经济 + 上下架 + 规则参数」。后台改 DB **变不出新 sprite、变不出新校验类型** —— 它只在「引擎已提供的键空间 / 校验原语 / 指标种类」内编辑。
- **每次写入**:过校验器 → 版本化 → 审计(谁/何时/改了什么)→ 可回滚。
- **形态(待定,见 §11.4)**:倾向**独立 admin web 页**(后端起一个 `/admin` 受保护路由 + 一套 `/api/admin/*`),与 Flutter app 分离。

## 11.1 三大模块

### 11.1.1 任务配置(task config)

> 依赖 [§7](07-gamemode.md) 的任务系统(**目前未实现**)。任务配置后台**只有在 §7 任务模型落地后才有意义** —— 否则没有「任务」这个实体可配。本节定义后台要能配什么,并据此反推 §7 任务模型要长成什么样。

一个**任务模板** = `{id, 标题/文案, trigger(何时出现/激活), validation(如何判定完成), reward(完成给什么), schedule(一次性/每日/可重复), enabled, season?}`。

- **trigger(生效规则)** —— 任务何时对用户可见/激活。候选原语(**引擎需支持的类型,后台只组合不发明**):
  - 时间型:每日生成(daily-gen,§7.3)、固定日期/季节窗口。
  - 条件型:用户达到某状态才解锁(如「已点亮 3 领域」后出现进阶任务)。
  - 事件型:某 `completion_event` 触发后派生(如首次记账 → 解锁「记一周账」)。
- **validation(校验规则)** —— 如何判定任务完成。**这是最需要先定的部分**,因为校验类型决定工程量。候选原语:
  - **指标阈值**:复用 §10 的引擎指标(`capture_count` / `streak_days` / `domain_total:<domain>` / 某 skill 累计数…)达到阈值 → 完成。
  - **事件计数**:在任务窗口内累计 N 个指定类型的 `completion_event`。
  - **手动/外部**:后台或外部信号标记完成(后置)。
  - 铁律:**新增一种校验类型 = 工程改 `completion.py` 的判定**;后台只能用已有原语 + 填参数(阈值、领域、窗口)。
- **reward(奖励)** —— 完成发什么。引用**组件库**(§11.1.2)里的 cosmetic_id(发到 `unlocked`),未来可扩展(岛元素 / 额度 / premium)。复用现有 `emit_completion_event` → 发奖通道。

> **开放问题(§11.4)**:§7 任务到底是「系统按规则自动派生」还是「后台手工编排的活动」?两者校验/生效模型不同。需先和你对齐 §7 任务形态。

### 11.1.2 组件库(component library)

> = §10.1「装饰目录 + 掉落池」做成可编辑后台。**关键是 §10.0 的「参数化 vs 形状」二分必须在 UI 上讲清,否则会误以为「什么组件都能后台新增」。**

每个组件一行:`{id, slot(skin|emblem|head|leftItem|rightItem|carrier|aura), 形/色, name(中文), rarity(普通|稀有|史诗|传说), drop_weight(掉落概率权重), premium(0/1), enabled(上下架), unlock_rule?(由哪个里程碑/任务发), season?}`。

- **增删的真实语义(必须在后台明示)**:
  - **参数化组件**(`skin` 体色 / `aura` 光环 / `emblem_color` 徽色 = 一个颜色值)→ **后台可真·新增**:填 `{id, name, 颜色 ramp}`,引擎按值上色,**无需美术**。删除 = 下架(`enabled=0`,不物理删,保护已拥有用户)。
  - **形状组件**(`emblem` 徽记 / `head` 头部 / `item` 双手 / `carrier` 承载 = 一段 sprite 画法)→ 后台可改**元数据/稀有度/掉率/上下架/由谁解锁**,但**新增一个新形状仍需工程 + 美术在 `mascot.js` 加画法**;后台「新增形状」按钮应引导走「工程工单」而非凭空建。
  - v3 的「徽记 = 颜色烘焙进件的命名组件」(金星/赤焰闪电…)在后台是 `形(bolt)×色(coral)` 的组合行;一个形可多配色行。
- **概率(probability)**:
  - 现状:`roll_drop` 是「从未拥有的可掉键里**均匀**随机」+ 全局 `_DROP_CHANCE=0.55`。要「配概率」需把掉落改为 **drop_weight 加权**(或按 rarity 权重),这是 §10.1.2 掉落池的工程改动 —— **后台配的是权重表,引擎按权重抽**。
  - lock-gated 组件(里程碑/任务发)不进随机池,由 `unlock_rule` 决定 —— 后台配的是「达成什么 → 发这件」。
- **校验器(§10 Stage 1 产物)**:每个 `id` 必须在引擎键空间内;掉落池只引用 enabled 键;参数化组件的颜色合法。任一不过 → 拒绝保存(不让裂图上线)。

### 11.1.3 用户总览(all-users view,只读)

> 直接回应「看所有用户的任务和奖励发放」。**只读**,不改用户数据。

- **用户列表**:user_id / 名字 / 宠物是否孵化 / 接住数 / 连续天 / 点亮领域 / 已拥有组件数。
- **单用户详情**:
  - 宠物当前装扮 + `unlocked`(已拥有组件,按 slot 分)+ `milestones`(累计计数)。
  - **奖励发放时间线**:该用户的 `completion_events` feed(什么动作 → 触发了什么掉落/解锁),即「奖励发放」的可见化 —— 这是目前**完全看不到**、最该先有的视图。
  - 任务进度(§7 落地后):进行中/已完成任务 + 各自校验进度。
- 价值:概率掉落 + 门控解锁现在是黑盒,这个视图让你能核对「到底发了什么、为什么」,也是调平衡(改 §11.1.2 权重/阈值)的依据。

## 11.2 架构(草案,待讨论)

- **数据模型**:沿用 §10 Stage 2 的 `game_config`(版本化行 + 生效时间 + 作者 + 可回滚)承载组件库/掉落池/里程碑/任务模板;用户态读现有 `pets` + `completion_events`(只读聚合)。
- **API**(全 admin-only,独立鉴权):
  - 组件库/掉落/里程碑/任务:`GET/PUT /api/admin/game-config`(+ `rollback`),复用 §10.3。
  - 用户总览:`GET /api/admin/users`(分页列表)、`GET /api/admin/users/{id}`(详情 + events feed)。**只读**。
- **前端**:一个极简内部 admin web(非 Flutter app);可先用后端直出 HTML + 少量 JS(同 report viewer 思路),不引重框架。
- **鉴权**:独立 admin 角色(env 白名单 user_id / 单独的 admin token);**绝不复用普通用户登录态**;所有写操作审计落库。

## 11.4 待讨论的开放问题

1. **任务形态**:§7 任务是「系统按规则自动派生(daily-gen)」还是「你在后台手工编排的活动/运营任务」?还是两者都要?这决定 trigger/validation 模型。
2. **校验原语集合**:任务完成判定要支持哪些类型(只指标阈值?要不要事件计数/手动标记/外部信号)?每多一类 = 工程量。
3. **组件「增删」的预期**:你说的「增加和删除组件」,是否接受「颜色类可真新增、形状类新增要美术工单」这个边界?还是希望先备一批形状 sprite 让后台从中上架?
4. **概率模型**:要「每件配概率」就得上 drop_weight 加权(改 `roll_drop`)。确认要做加权,还是先「按稀有度档位配权重」即可?
5. **后台形态与鉴权**:独立 web admin 可接受?admin 账号怎么定(env 白名单 / 单独后台账号体系)?是否要多人 + 审计?
6. **范围与时机**:beta 阶段是否先只做 **11.1.3 用户总览(只读)** —— 成本最低、立刻让奖励发放可见;组件库/任务配置等 §10 Stage 1 + §7 落地后再上?
