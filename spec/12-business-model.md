## 12.1 成本分析（基于真实 prompt + 管线,非生产遥测）

> ⚠️ **重要前提**:代码里**没有 token 用量日志**(LiteLLM 的 `response.usage` 没人记)。以下是**按真实 prompt 体量 + 管线形状**算的工程估算,**非生产实测**。先加日志(§12.5)再校准。

**测量到的(真实):**
- **模型** = `deepseek/deepseek-chat`(DeepSeek V3)**直连 API**(`api.deepseek.com`,国内托管),flash / chat / report 都用它(`core/llm.py`)。
- **费率**(**以 DeepSeek 官方直连价为准**,见 platform.deepseek.com):输入 **$0.27/M**(miss)/ **$0.07/M**(cache hit);输出 **$1.10/M**。直连=DeepSeek 原生自动缓存,cache hit 可靠(不再经 OpenRouter 路由)。
- **prompt 体量**:flash dispatcher **10.3k 字**、flash 子技能 2.9–8.3k 字、chat 指令 **20.9k 字**(每轮都发,20 条历史窗口)、report dispatcher **8.9k 字**、report 内容技能 2.7–3.5k 字(**无工具,数据预注入,单次调用**)。
- **管线形状**:**flash = 1 dispatcher + N 子技能**(N=意图数,每子技能 ≈2 个 round-trip:工具调用 + 收尾);**chat = 1 次大调用**(+1 RT 若做 CRUD);**report = dispatcher + 内容技能**(渲染是确定性 Python,**零 LLM**)。
- 换算假设:中文为主 ≈ **0.5 token/字**;工具 schema ≈ 2k token/含工具调用。

**每次请求成本(估):**

| 请求 | 输入 token | 输出 token | 未缓存 | **缓存** |
|---|---|---|---|---|
| Flash(典型 ~1.3 意图) | ~19,000 | ~450 | 0.56¢ | **0.30¢** |
| Chat(典型) | ~20,000 | ~350 | 0.58¢ | **0.30¢** |
| Report | ~12,500 | ~2,650 | 0.63¢ | **0.45¢** |
| 洞察 / 周更 | ~4,000 | ~150 | 0.12¢ | **0.08¢** |

- **反直觉发现:报告很便宜(~0.5¢),不是重负担。** 因内容技能**无工具 + 数据一次性预注入**(§6 设计),report = 一次 dispatcher + 一次内容调用;成本在它 ~2.6k 的输出,不在工具循环。→ **报告可以慷慨给。**
- **洞察基本免费**(~0.08¢/次);1 周 1 次的 quota 是为了体验/不吵,**不是成本考量**。

**每个免费用户的月成本(按下方提案 quota):**

| | 未缓存 | **缓存** |
|---|---|---|
| **典型用户**(~200 捕捉、~60 chat、1 报告) | ~$1.48 | **~$0.80** |
| **极限**(把每个 quota 用满) | ~$6.81 | **~$3.62** |

- 极限里 **捕捉占 ~75%**(900 次 × 多调用管线)→ **30/天 的捕捉上限是最重要的成本闸**,设它是对的。
- 典型免费用户 **≈ $1/月**;极限 ≈ $4–7/月。

**三个比任何 quota 都更影响成本的杠杆:**
1. **缓存 ≈ 2×**:代码里**没设 `cache_control`**;**DeepSeek 直连 = 服务端自动缓存,命中可靠**(切直连后不再有「经 OpenRouter 是否命中」的路由疑问)。静态大 prompt(10k+21k 字)是完美缓存目标。上表两列之差全在这。
2. **裁 prompt**:20.9k 字的 chat 指令每轮都发、10.3k 字 dispatcher 每次捕捉都发。**裁 30% = 每条请求省 30%**,免费/付费通吃。
3. **加用量日志(§12.5)**:三行代码记 `response.usage` → 两周后有**真实**数字 + 每用户成本看板。性价比最高的一步。

## 12.2 定价提案（Free + 单一 Pro）

**约束(决定什么模式不能选):** 成本几乎全在 LLM;但产品命题是**零摩擦随手记**——任何让用户**捕捉时犹豫**的收费(按次扣点 / 捕捉撞墙)会反噬核心习惯,且**和你自己的岛/球球(奖励多记)对着干**。所以排除「纯 credits / 按捕捉计费 / 抽卡」。

**把收费线画在「捕捉之外」:**

| | Free(慷慨) | Pro(单一订阅) |
|---|---|---|
| **捕捉**(语音/硬件 flash) | **30/天**(≤900/月),撞顶**排队不丢** + 软提示 | 不限(仍留防滥用日上限) |
| **Chat** | **300/月** | 不限 |
| **报告**(§6) | **~4/月**(成本低,可再放宽) | 不限 |
| **洞察 / 周更点评** | **1/周** | 更频繁 |
| 整理 / 库 / 时间线 / 领域 | 全开 | 全开 |
| 自定义技能 | ~3 个 | 到上限 30 |
| 已连接应用(外部 MCP) | — | ✅ |
| 球球**玩法解锁**装饰 | ✅(记录就能解,**非 pay-to-win**) | ✅ |
| **premium 外观**装饰(§9.7) | — | ✅ 打包进 Pro |

**为什么是这个:**
- **捕捉永远慷慨**(30/天对正常用户基本无感、给岛留空间),付费闸落在**贵的(无)+ 进阶(连接应用、技能数)+ 产物(报告/洞察更多)+ 外观**上。
- **单档**:用户只一个决定(Free/Pro),实现只一个 flag。
- **复用已拍原则**:捕捉数计量、外观只卖不锁功能、不抽卡、不 grind(§7.11)。

**价格含义(基于 §12.1):** 典型免费用户 ~$1/月、极限 ~$4–7/月 → **Pro 定 ~$7–9/月**,一个 Pro 覆盖 ~5–8 个典型免费用户或 ~1 个极限用户。**价格点 = 市场题,留 TODO。**

**更轻的过渡版(Stage 0):** 若上线初期用户量小,可先 **全功能免费 + 只卖 premium 外观**(摩擦最低、攒善意);量一上来切到上面的 Free+Pro。

## 12.3 护栏（承 [§7.11](07-gamemode.md)）

- **撞顶绝不吞用户的话**:捕捉到上限 → **排队 / 照存 + 软 upsell**,绝不丢(尤其硬件卡录的,用户默认已存)。
- **只卖外观,绝不锁功能**:核心动作永远免费可用;**玩法解锁**的装饰不拿去卖(否则变 pay-to-win,§9.7)。
- **不抽卡、不 FOMO、不 grind 换额度**。
- **数「次数」不数 token**:用户能理解「这月还剩 X 次」,理解不了 token。
